## 導言

Spotify 工程師 Niklas 在一場主題演講中分享了一個驚人的數據：

> 「過去兩週我準備這份簡報時，我把同一個數字改了三次——因為它還在一直漲。」

這個數字，是 PR 頻率的成長幅度。

2024 年 11 月，OpenAI 發布 OPUS 4.5 之後，Spotify 的工程團隊迎來了一個轉折點。此後每天，工程師與 AI Agent 共同產出的 PR 數量成長了 76%——而這個數字還在持續飆升。

這不是一個關於「用 AI 寫程式」的故事。這是一個關於**工程組織如何重新思考整個開發流程**的故事。

---

## 從 3,000 人工程團隊說起

Spotify 目前約有近 3,000 名工程師，運行在一個極端的架構組合上：

- 一個 **4,000 萬行程式碼**的後端單體倉儲（monorepo）
- 數千個**小型多元倉儲**（poly repos）
- 每天約 **4,500 次部署**到生產環境

多年來，Spotify 一直努力優化開發者體驗，確保工程師能夠快速、頻繁地將變更交付給用戶。但即使如此，代码库的增长速度仍然是工程师数量增长速度的 7 倍。

Niklas 直言：「工程師越來越多的時間，被用來維護現有代碼庫，而不是為用戶構建新價值。」

這不是 Spotify 獨有的問題——但 Spotify 選擇正面迎擊。

---

## FleetShift：用自動化改寫遷移遊戲

面對這個挑戰，Spotify 開發了一套內部工具稱為 **FleetShift**（fleet management 系統的一部分）。

過去的做法是這樣的：要升級 Java 版本，公司必須發送通知給數百個團隊，讓每個團隊自行執行遷移。一次升級往往需要**數月**才能覆蓋數千個元件。

Niklas 說：「那個時代，遷移是工程師反饋最高的痛點。」

FleetShift 的核心思路很簡單：**不要一個一個元件去做，而是把整個 fleet 當成一個整體來 mutate。**

他們建立了一個自動化框架，讓系統可以：
1. 自動生成遷移 PR
2. 自動在 CI 環境中驗證這些 PR 的安全性
3. 自動合併——**不需要任何人為介入**

到目前為止，Spotify 已經合併了 **250 萬個**這類自動化維護 PR，全部由系統自行完成，工程師完全不需要操心。

問題在於：簡單的設定檔變更依賴腳本可以處理，但一旦涉及複雜的 API 替換，腳本就會變得極度複雜。

Niklas 引用了 Google 工程師 Hiram 的發現：「當你寫一個遷移腳本，並在數百萬行代碼、數千個元件上運行時，你會遇到**每一個**角落案例。」

這就是為什麼——當 LLM 出現時，Spotify 找到了新的方向。

---

## Honk：讓 Claude 替你遷移

經過多次迭代，Spotify 開發出了一個內部工具叫做 **Honk**。

Honk 底層使用 Claude（透過 Agent SDK），包裝在 Kubernetes pod 中，可以大規模排程執行。Claude 被給予一組信任的工具，其中包括在 CI 環境中運行構建的能力——因為 Spotify 的用戶端運行在多個不同的作業系統上，這點極為重要。

工作流程是這樣的：

- **FleetShift** 負責排程和協調——決定哪些倉庫需要執行什麼遷移
- **Honk** 在中間負責實際的代碼變更

對於一個典型團隊而言，登入 FleetShift 面板可以看到：哪些 PR 已建立、哪些已合併、哪些在 CI 中失敗需要人工處理。

Niklas 分享了一個具體案例：「過去需要數百個團隊執行數週的 Java 升級，現在只需要一個工程師，**三天**就完成了。」

這套工具現在已經商業化，透過 Spotify 的 Backstage 開發者入口網站提供給其他企業使用。

---

## 從 Slack 呼叫 Agent：Honk 的進化

但開發者的創意不止於此。

很快，工程師們發現：既然 Honk 能處理遷移任務，有沒有可能透過 Slack 直接叫用它？

答案是可以的。現在 Spotify 內部，團隊可以在 Slack 對話中直接 @Honk，Honk 就會去執行任務，完成後回來附上一個 PR。

Niklas 說：「這變成了一種非常常見的工作模式——Slack 討論到一半，@Honk，然後它就去做事了。」

昨天（演講前一天），Spotify 發布了 **Honk V2**（內部其實是第八版）。這個版本將 Honk 整合進了 Spotify 的 Agent 編排工具 **CHERP**，讓多人可以在同一個 Agent session 中協作——就像 Google Docs 但適用於 Claude。

Niklas 個人最興奮的功能：**多人協作**。一個工程師不再單獨面對一個 Agent，而是可以與團隊成員共同使用同一個 session，給予回饋與方向。

---

## 讓代碼庫為 Agent 做好準備

這是 Niklas 演講中個人認為最值得深思的部分。

多年來，Spotify 有一個核心信念：**使用的技術越少，我們走得越快。**

原因很直接：
- 當團隊對同一組技術有深度掌握，能構建更好的產品
- 減少團隊的技術選擇疲勞
- 代碼庫結構趨於一致，讓跨團隊協作更容易

一個典型的 Spotify 後端服務，使用相同的技術棧、相同的設計模式。這不只對人類工程師有好處——**對 Agent 也同樣有效**。

Niklas 的觀察是：「如果 Claude 看到的其他代碼結構相當一致，Claude 的表現會更好。我們確實有數據證明，在更碎片化的代碼庫中，Claude 的表現更差。」

這個發現引出了 Spotify 的另一個核心工具：**Backstage**。

---

## Backstage：不只是人類的开发者入口

Backstage 是 Spotify 的開發者入口網站，最初解決的是一個很基礎的問題：當發生事故時，知道某個元件的負責人是誰。

演講前，Spotify 的工程師需要使用大約 **100 種不同工具**——一個查部署、一個查 CI、一個查 A/B 測試。Niklas 坦言：「那些工具都很爛。」

Backstage 的起點就是一本目錄：哪些元件、誰負責、怎麼聯繫。

多年下來，它已經擴展成一個完整的開發門戶。今天，Spotify 工程師幾乎所有與軟體元件相關的動作，都在 Backstage 中完成。

而對於 Agent，Backstage 的價值同樣巨大：Claude 可以透過 Backstage 查詢元件負責人、在 Slack 上 ping 相關團隊、存取各種工具。這讓 Agent 能在真實的組織上下文中有意義地行動。

透過所謂的 **Golden Path**（Spotify 的技術標準化框架）和 **SoundCheck**（團隊自我評估介面），Spotify 確保代碼庫的一致性——而這些標準化同樣適用於 AI Agent。

Niklas 說：「當 Claude 在我們的代碼庫中運行時，如果它使用了我們知道不是最優的基础設施呼叫方式，lint 系統會立即回饋給它，讓它糾正自己。」

這是一種讓 lint 規則同時服務人類工程師和 AI Agent 的巧妙方式。

---

## 結語：約束正在轉移

Niklas 的演講尾聲，給出了一個清晰的總結：

**強工程實踐的需求，並沒有因為 Agent 的到來而消失。** 測試充分的代碼庫、能被 Agent 呼叫的測試——這些是讓 Agent 更自主、產生更好解決方案的前提。

同時，Human Judgment 比以往任何時候都更重要——只不過現在的問題是：**在哪裡投入人類的判斷力？**

Niklas 提到一個現象：PR 數量增加了 76%，但這也意味著有 76% 更多的 PR 需要人類審查。「開發者最常見的反饋是：PR 多到審不過來。」

Spotify 的回應：已經在對部分 PR 進行 order-proving，識別為安全的直接合併，將人類審查集中在真正重要的地方。

而更深層的轉變是：**程式設計不再是瓶頸了。**

過去產品開發生命週期中，最大的等待是工程師實作功能。現在，這個約束正在鬆動。

Niklas 舉了一個極端的例子：「現在 Spotify 任何人都可以在客戶端單體倉儲中叫用 Claude，透過一組技能和基礎設施，讓 Claude 構建任何你想嘗試的功能。幾分鐘後，你就能得到一個可以安裝在設備上的 App，分享給同事測試。」

這將過去需要數天或數週的原型開發，縮短到**分鐘級**。

連 Spotify 其中一位執行長（Niklas 只說「one of our CEOs」，未指名）現在也在用這套工具构建原型。

---

這個故事的結論，也許並不是「AI 將取代工程師」這麼簡單。

而是：當**程式設計**不再是瓶頸時，人類判斷力、决策质量、优先级排序——這些真正稀缺的能力，正在變得比以前更重要。

約束在轉移。聰明的組織，已經在為下半場做準備。
