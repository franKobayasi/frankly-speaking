## 1. Claude Code 撰寫最佳實踐

### 對話篇

#### 核心原則

**清晰、具體、可操作**

#### 有效的提問方式

| 類型 | 示例 | 說明 |
|------|------|------|
| 具體請求 | 「用 React 寫一個登入表單，包含驗證」 | 明確技術棧與功能需求 |
| 範圍限定 | 「用 Python 解析 CSV，不要用 pandas」 | 指定實現方式 |
| 目標導向 | 「寫一個 API 文件生成器，輸出 Markdown」 | 說明最終產出 |

#### 應避免的提問

- ❌ 模糊描述：「做個好玩的東西」
- ❌ 缺少上下文：「幫我優化這段程式碼」
- ❌ 過於開放：「給我一些建議」

#### 對話技巧

1. **分步驟進行**：複雜任務拆成多個小步驟，逐步完成
2. **提供範例**：展示期望的輸出格式或程式碼風格
3. **及時確認**：關鍵决策點主動徵求用户意見
4. **明確截止條件**：說明何時算完成、評估標準是什麼

#### 程式碼生成原則

```typescript

// -  timeout 為 10 秒
class HttpClient {
  async request(url: string, options?: RequestOptions): Promise<any> {
    const controller = new AbortController();
    const timeout = setTimeout(() => controller.abort(), 10000);
    // ...
  }
}
```

#### 重構與優化

1. **先理解再修改**：閱讀現有程式碼架構再提出改動
2. **保留原有行為**：重構不應改變功能 semantics
3. **小步快跑**：大規模改動分多次進行，每次可逆
4. **添加測試**：重構前先確保有足夠的測試覆蓋

#### 程式碼審查重點

- 錯誤處理是否完整
- 資源是否正確釋放（記憶體、連線、檔案）
- 邊界條件是否考慮
- 命名是否清晰

### 任務篇

#### 任務分類與策略

| 任務類型 | 策略 | 範例 |
|----------|------|------|
| 單次任務 | 直接執行 | 「翻譯這段文字」 |
| 迭代任務 | 定期回顧 | 「Review 每個 PR」 |
| 複雜專案 | 分階段規劃 | 「建立電商系統」 |
| 研究任務 | 先調査再行動 | 「.compare React vs Vue」 |

#### 有效的任務管理模式

1. **明確範圍**：界定「什麼在範圍內，什麼不在」
2. **設定里程碑**：將大任務拆成可驗證的小階段
3. **預期管理**：說明預計耗時與可能遇到的阻礙
4. **定期同步**：複雜任務保持沟通頻率

---

### 對話框架篇

#### 多輪對話模式

```
用戶 → [明確任務] → AI → [確認理解] → 用戶 → [補充資訊] → AI → [執行] → 用戶 → [回饋] → AI → [優化]
```

#### 對話節奏控制

- **開頭**：確認任務範圍與目標
- **中間**：彙報進度、請求关键决策点指示
- **結尾**：總結完成事項、列出後續步驟

#### 常見對話模式

| 模式 | 適用場景 | 範例 |
|------|----------|------|
| 教學模式 | 學習新技術 | 「教我 Rust 的所有權系統」 |
| 合作模式 | 共同開發 | 「我們一起重構這個模組」 |
| 審查模式 | Code Review | 「檢查這段程式碼的問題」 |
| 諮詢模式 | 決策建議 | 「這個架構選擇好嗎？」 |

## 2. Agents 撰寫最佳實踐

### Agent 基礎篇

#### 為什麼使用 Agent？

Agent 是能夠自主完成複雜任務的 AI 系統，具有以下特性：

- **自主决策**：能根據狀況選擇下一步行動
- **工具使用**：可調用外部工具（API、檔案系統、指令列）
- **長期記憶**：跨會話保持上下文
- **目標導向**：围绕明确目标規劃行動

#### 設計 Agent 的核心要素

```
Agent = LLM + 工具 + 記憶 + 規劃能力
```

| 要素 | 說明 | 範例 |
|------|------|------|
| LLM | 推理與生成核心 | Claude Sonnet 4 |
| 工具 | 擴展能力邊界 | 檔案读写、命令執行、API 呼叫 |
| 記憶 | 保持上下文 | 對話歷史、向量資料庫 |
| 規劃 | 分解與執行 | Chain of Thought、ReAct |

---

### 對話框架篇

#### Agent 對話設計原則

1. **明確角色**：定義 Agent 的身份與能力邊界
2. **清晰目標**：用戶目標轉化為可執行的子任務
3. **結構化輸出**：定義 Agent 輸出格式以利解析
4. **錯誤處理**：預設各種失敗情況的應對方式

#### 對話流程設計

```
[User Input] → [Intent Classification] → [Entity Extraction]
                                            ↓
                          [Task Planning] → [Tool Selection]
                                            ↓
                          [Execution] → [Response Formatter] → [User]
```

#### 多 Agent 協作模式

| 模式 | 描述 | 適用場景 |
|------|------|----------|
| 主從模式 | 一個主 Agent 協調多個子 Agent | 複雜系統開發 |
| 平行模式 | 多個 Agent 同時處理不同任務 | 資料處理、批量作業 |
| 串聯模式 | Agent 輸出作為下一個 Agent 輸入 | 多步驟工作流 |

---

### 複雜任務篇

#### 任務分解策略

```
複雜任務 → 子任務 1 → 子任務 2 → 子任務 3 → 整合結果
     ↓           ↓           ↓           ↓
  獨立執行   依賴管理    狀態追蹤    錯誤恢復
```

#### 狀態管理

1. **追蹤進度**：記錄每個子任務的完成狀態
2. **保存上下文**：確保中斷後能從上次位置繼續
3. **處理失敗**：子任務失敗時決定重試、放棄或降級

#### 驗證與品質控制

- **逐步驗證**：每個子任務完成後進行檢查
- **最終確認**：整體完成後做整體性驗證
- **用戶回饋**：設計回饋機制，持續改進

## 3. SKILL 撰寫最佳實踐

### 基礎原則篇

#### 什麼是 SKILL？

SKILL 是 Claude Code 中的可自定義指令集，讓使用者能夠：

- 將重複性工作自動化
- 建立個人化的 AI 互動模式
- 封裝專業領域知識
- 跨會話保持一致的工作流程

#### SKILL 結構

```yaml
name: skill 名稱
description: 簡短描述功能
trigger: 觸發條件（關鍵字或模式）
actions:
  - 執行動作 1
  - 執行動作 2
```

#### 設計原則

| 原則 | 說明 | 範例 |
|------|------|------|
| 单一职责 | 每個 Skill 專注一件事 | 「翻譯」skill 只做翻譯 |
| 可複用性 | 設計成通用模組 | 參數化輸入輸出 |
| 清晰介面 | 明確定義輸入輸出 | 固定的參數格式 |
| 錯誤處理 | 内建失敗處理機制 | timeout 重試邏輯 |

---

### 進階應用篇

#### SKILL 组合使用

```
Skill A (提取) → Skill B (處理) → Skill C (輸出)
```

#### 最佳實踐

1. **從簡單開始**：先建立基礎 Skill，再逐步擴展
2. **文件化**：為每個 Skill 撰寫使用說明
3. **版本控制**：追蹤 Skill 的變更歷史
4. **測試驗證**：確保 Skill 在各種情況正確運作

#### 進階技巧

- **條件判斷**：根據上下文動態選擇執行分支
- **迭代處理**：對清單項目逐一處理
- **狀態保存**：跨次對話記憶偏好設定
- **鉤子 Hook**：在特定事件觸發自訂行為

#### 維護與優化

- 定期檢視 Skill 的使用頻率與效果
- 收集使用者回饋持續改進
- 保持 Skill 的簡潔與可理解性

---

## 總結

有效運用 AI 協作工具的關鍵在於：

1. **清晰溝通**：提供具體、明確的指示
2. **適當分工**：了解何時該自己動手、何時該委託 AI
3. **持續學習**：不斷優化互動方式，追求更好的協作效率
4. **工具整合**：善用 Agent 與 SKILL 將重複任務自動化

遵循這些最佳實踐，你將能充分發揮 AI 的潛力，提升工作效率與品質。
