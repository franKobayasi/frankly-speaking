## 導言：你的 AI Agent 是你的工具，還是學徒？

想象你走進一間馬具工坊。老師傅遞給你三種不同的馬具：

第一種是**現成的韁繩**——拿來就能用，但每次騎馬都要你親手操控方向。
第二種是**會調節的輓具**——可以根據馬的反應自動調整鬆緊，但本質上還是服從你的命令。
第三種是**會學習的馬鞍**——它會記住這匹馬的步態、脾氣、偏好，騎得越久，貼合度越高。

大多數 AI Agent 屬於前兩種。你每次對話都要重新解釋上下文，每次新 session 都是一張白紙。規則需要你維護，偏好在關閉分頁後就消失。

**Hermes Agent** 屬於第三種。它的設計目標不是成為更聽話的工具，而是成為**會觀察、會歸納、會自我改進的學徒**。

這個定位，值得你花十分鐘深入了解。

## 馬具比喻：三種工具有三種不同的「陪伴關係」

在比較功能之前，我想先建立一個概念框架，幫助你理解這三個工具真正的定位差異。

我把 AI Agent 想像成一種「數位陪伴關係」——你與工具之間的互動模式，決定了它能為你做什麼。

**Claude Code** 像一把**隨取即用的專業工具**。你需要它時，它立刻上線；你不需要時，它安靜待命。每次對話都是嶄新的開始，沒有歷史包袱，適合需要即時、獨立、專案範圍清晰的任務。

**OpenClaw** 像一個**可配置的自動化框架**。你定義規則與流程，它幫你執行。你是導演，它是劇團——每次演出你可以選擇不同的劇本（不同的 Agent 配置），團隊協作與流程控制是它的強項。

**Hermes Agent** 像一個**會成長的學徒**。你帶它做過一次專案，它就會記住流程；你糾正過一次錯誤，它就不會再犯。它上線時間越長，對你和你的工作方式的理解就越深——這也是它與前兩者最根本的差異。

**這三種關係並不競爭，而是服務不同的需求場景。** 就像你會需要螺絲起子也會需要鋸子，專業工作者的工具箱裡本來就不只一把工具。

## Hermes Agent 的核心創新：內建的學習迴圈

Hermes Agent 由**Nous Research**（專注於語言模型研究的實驗室）開發，於 2026 年 2 月开源發布。它的核心賣點不是更强的模型，而是**內建的自我學習迴圈**——這讓它與市場上其他 Agent 工具产生了本质的区别。

### 學習迴圈如何運作

每次完成任務後，Hermes 會自動執行一次「自我檢討」：

1. **我應該記住什麼？** → 有價值的資訊寫入長期記憶
2. **這是重複的模式嗎？** → 自動生成 Skill 檔案（相當於工作流程的 SOP）
3. **使用者糾正我了嗎？** → 更新對應的 Skill，下次不再犯同樣的錯誤

這個過程**完全自動觸發**，不需要你下任何命令。

想象一下：第十次請它幫你寫 Python 爬蟲，它會記得你偏好 httpx 而非 requests、日誌寫到檔案而非終端機、變數命名要用底線分隔而不是駝峰式。這些都不是你教的——是它從觀察中自己歸納出來的。

### 三層記憶系統

Hermes 的記憶架構分為三層，設計邏輯是**按需檢索，而非全量載入**：

| 層級 | 儲存形式 | 作用 |
|---|---|---|
| **Session 記憶** | SQLite + FTS5 全文檢索 | 對話歷史完整保存，可依主題快速檢索 |
| **持久記憶** | 使用者偏好檔案 | 自動載入每個新對話，不需要重新提示 |
| **Skill 記憶** | Markdown 技能檔案 | 記錄「如何正確做事」——經過驗證的工作流程 |

這個設計的好處是：即使使用三個月後的對話，檢索速度與三天後的體驗一樣快速——因為它不是把所有歷史塞進 context window，而是只抓取與當前任務相關的內容。

### 會自我進化的 Skills

Hermes 的 Skill 並非靜態設定檔，而是不斷生長的知識庫。來源有三：

| 來源 | 說明 |
|---|---|
| **內建** | 安裝時附帶 40+ 個預置 Skills |
| **Agent 自創** | 從重複任務中自動歸納生成 |
| **社群 Hub** | 從 agentskills.io 一指令安裝 |

**殺手級功能：Skill 會在使用中自我改進。** 每當你給予具體回饋（「下次先檢查表格是否存在」），Hermes 會修改對應的 Skill 檔案。下一次執行時，改進自然內化。

Skills 遵循 **agentskills.io 開放標準**，理論上可以跨工具攜帶——你為 Hermes 寫的 Skill，原則上可以用在其他支援該標準的工具上。

## 功能版圖：47 個工具 + 15+ 訊息平台

除了學習機制，Hermes 的功能覆蓋範圍也值得注意：

**47 個內建工具**，涵蓋執行、資訊、媒體、記憶、協調五大類別。加上對 **MCP（Model Context Protocol）** 的支援，原則上可以連接超過 6,000 個外部應用——GitHub、Slack、Jira、資料庫、Google Drive 都包含在內。

**多平台 Gateway** 是另一個差異化特點。單一进程同時支援 Telegram、Discord、Slack、WhatsApp、Signal、Email、Matrix、Mattermost、Home Assistant 等 15+ 平台。

**實際意義是什麼？** 在通勤的 Telegram 上開始一個任務，到辦公室後在終端機無縫繼續。Hermes 不區分訊息來自哪個平台——同一個大腦、同一份記憶、同一組 Skills。

## 實際應用場景：什麼情境適合選用 Hermes？

### 場景一：需要 24/7 後台自動化

Hermes 的 Cron 排程 + 多平台 Gateway 讓它可以在背景執行長期任務：

- 每早 8 點主動發送伺服器健康報告到 Telegram
- 每晚自動拉取 GitHub 更新、生成 code review 摘要
- 監控異常日誌，符合條件時主動 Alert 到指定頻道

這個場景是 Claude Code 和 OpenClaw 較難覆蓋的——後兩者更偏向「按需求啟動」的模式，而非持續運行。

### 場景二：長期專案的上下文累積

如果你的專案為期數月，中間會有間斷，Hermes 的三層記憶系統可以幫你保持連續性。每次回到專案，不需要重新解釋團隊的 coding style、偏好工具、已知問題——這些都已經寫進它的記憶。

### 場景三：跨平台工作流

在 Discord 上交代背景，出門後在 Telegram 繼續討論，到公司直接在終端機查看執行情況。記憶和上下文全程無縫接續——這對於需要多點移動、多裝置切換的工作者特別有價值。

## 需要注意的限制

**1. 記憶不會自動過期**
SQLite 資料庫只會越寫越大。需要定期檢查 `~/.hermes/` 清理不需要的歷史，否則磁碟空間和檢索效能都可能受到影響。

**2. 有效的回饋需要具體**
「這個不太對」這類模糊回饋不會產生任何改變。需要具體說明問題所在，Hermes 才能有效更新 Skill。這其實對使用者提問的能力提出了一定要求。

**3. 自動學習伴隨控制權讓渡**
當系統自己會歸納規則，你就需要偶爾打開 Skill 檔案檢查一下，確認自動生成的規則仍然符合預期。長期下來，這是「省力」還是「多了一層維護負擔」，取決於你的使用情境。

**4. 社群生態仍在早期**
相較於 OpenClaw 的 ClawHub（擁有 5,700+ Skills 的網路效應），Hermes 的預置 Skills 數量和多樣性目前不及競爭對手。如果你是那種「不想自己寫 Skill，想直接用別人寫好的」的使用者，這個階段可能需要更多自行摸索。

## 安裝方式

一行指令，無需任何前置條件：

```bash
curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash
source ~/.bashrc
hermes version  # 驗證安裝
```

支援 Linux、macOS、WSL2。也可透過 Docker 執行：

```bash
docker pull nousresearch/hermes-agent:latest
docker run -v ~/.hermes:/opt/data nousresearch/hermes-agent:latest
```

## 結語：工具沒有最好，只有最適合

Hermes Agent 最大的創新不是任何單一功能，而是一個概念：**把「學習」從人類的責任轉移到了系統本身**。

傳統的 Agent 設計把人放在控制環路中心——你要寫規則、維護上下文、糾正錯誤。Hermes 的設計把人放在**元層級**——你只需要偶爾審查和Override，系統自己會朝正確方向累積。

這個方向是否正確，需要時間和社群驗證。但對於需要長期運行、跨 session 累積上下文、24/7 後台自動化的使用者，Hermes Agent 提供了一個目前市面上獨一無二的解決方案。

**選擇工具的邏輯，其實很簡單：**

- 需要即時代碼建議 → Claude Code
- 需要透明的團隊協作框架 → OpenClaw
- 需要會成長的長期學徒 → Hermes Agent

三種需求不一定互斥，你的工具箱裡可以同時有這三把工具。
