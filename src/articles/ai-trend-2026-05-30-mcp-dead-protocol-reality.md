上週，工程社群被一個標題點燃：「MCP is dead?」（MCP 已死？）。作者 Quandri 團隊基於自身產品實測，提出三個具體的技術控訴：context window 損耗過高、營運可靠性低落、以及與既有 CLI/API 功能的重疊。

這篇文章在 Hacker News 獲得 62 點、54 則留言，並且在兩小時內引來了 OpenAI 負責 ChatGPT App Store、Codex plugins 與所有 MCP 事務的團隊成員親自回覆。這種層級的「官方對官方」對話，在開放社群中並不常見。

## MCP 的技術控訴：數字說話

Quandri 的批評不是主觀感受，而是基於他們實際堆疊的測量數據。

他們測量了自家 AI 代理日常使用的四個 MCP 伺服器——Linear、Notion、Slack、Postgres——發現**77 個工具定義就消耗了約 84,308 字元（約 21,077 tokens）**。對照各家模型的 context window，這不是一個可以忽略的數字：

| 模型 | Context Window | MCP 工具定義佔比 |
|------|---------------|-----------------|
| Claude (200K) | 200,000 tokens | 10.5% |
| GPT-4o (128K) | 128,000 tokens | 16.5% |

一個還在發展中的 AI 代理產品，光是「讓工具可以被調用」就已經吃掉了 10-16% 的 context——而且這是**靜態消耗**，無論當前任務是否用到這些工具。

除了 context 消耗，Quandri 還列出營運可靠性的問題：初始化失敗需要重新認證、MCP 伺服器进程崩潰導致工具在中途失效、以及工具權限的不透明。**「這不是某個特定 MCP 伺服器的問題，這是架構上的問題。」** 他們寫道。每一層 MCP 伺服器，都在 LLM 和底層 API 之間新增了一個進程層，而這個進程層自帶的複雜性與不穩定性，是無法透過更好的實作完全消除的。

## CLI 的反擊：為何把同樣的任務交給 CLI 更有效

更有趣的是第三個控訴：MCP 與既有的 CLI/API 解決方案存在大量重疊，而且 CLI 在多個維度上表現更好。

Quandri 的比較很直接：

| 維度 | CLI / API | MCP |
|------|-----------|-----|
| 人機一致性 | 人類和 LLM 使用相同命令 | 只存在於 LLM 對話中 |
| 可組合性 | 可以用 pipe、jq、grep 自由組合 | 被鎖在伺服器的回傳格式裡 |
| 可除錯性 | 在終端機立即重現 | 只能在對話語境中重現 |
| 訓練資料 | 已有 man pages、StackOverflow 學習 | 需要另外寫工具定義 |
| 安裝成本 | 多數已預設安裝 | 需要伺服器架設、認證、进程管理 |

以查詢一張 Linear 票券為例：CLI/GQL 的方式是讓 LLM 生成一個 curl 指令，這個指令本身只需要極少的 tokens，且執行結果可以被人類工程師立即驗證。但 MCP 的方式是**永遠把所有工具定義載入**，無論當前任務是否需要，這是一個架構上的必然損耗。

## OpenAI 的回擊：MCP 作為「API 品牌」的勝利

面對這些批評，OpenAI 團隊成員 mxstbr 的回覆反而是最值得注意的觀點。他的核心論點是：**「MCP 是否作為傳輸協定使用，完全不重要。重要的是幾乎每家公司都在建立 MCP 伺服器。」**

他的邏輯是這樣的：許多公司沒有 CLI，更別說外部 API。當 AI 代理的時代來臨，這些公司不想要被排除在外——所以他們選擇建立 MCP 伺服器，因為這是「LLM 可以使用的 API」的代名詞。**「這讓我們的 AI 代理可以存取它們原本永遠無法觸及的服務。」** mxstbr 寫道。

從這個角度看，MCP 的價值不在於它是最好的技術方案，而在於它填補了一個市場空白：當「讓 AI 代理存取你的服務」變成一種剛性需求，大量的公司不會去評估各種技術選項，他們只會選擇一個已經被主流模型支援的標準。

HN 留言中另一位開發者 ford 的補充更精準：「MCP 本質上不過是『LLM 可用的 API』的代名詞。這表示更多的服務正在建立 API——那些過去對技術不太積極的公司，不想在人人使用代理的時代被淘汰。」

## 這場爭論的真實本質

表面上這是一場技術爭論，但深入觀察就會發現，這其實是**兩種 Agent 設計哲學的路線之爭**。

**第一種路線**把 AI 代理視為「有代理能力的使用者」——它應該使用和人類相同的工具，透過 CLI、API 與系統互動。這派認為，額外的抽象層只會帶來不必要的複雜性與損耗。

**第二種路線**把 AI 代理視為「新物種」——它需要專門為它設計的介面，與人類使用的工具有所區隔。這派認為，MCP 的價值正是它**不是** CLI——它是專門為 LLM 設計的。

一個具體的訊號：mxstbr 在回覆中提到，Codex 的 computer 與 browser use 功能「讓這個論點變弱了很多」。如果 AI 可以直接操作電腦和瀏覽器，那麼「存取服務」的命題確實會被重新定義。但這並沒有否定 MCP 的價值——它只是把 MCP 的應用場景收斂到更特定的需求上。

## 展望：未來三個月的觀察重點

這場「已死/未死」的爭論，短期內不會有結論。但有幾個觀察方向值得持續追蹤：

**第一，「MCP 還是 CLI」的選擇會隨場景分化。** Quandri 的文章已經提出一個具體框架：本地開發、個人用途，CLI/Skills 更輕更快；生產環境、共享團隊的資料庫，MCP 的安全guardrails 有其必要。這個分裂預示著未來的生態不会是「一個標準統一天下」，而是根據使用場景形成不同的最優解。

**第二，Context Window 的軍備競賽會影響這個爭論的走向。** 如果模型的 context window 繼續以每年 2-3 倍的速度成長，今天看起來嚴重的 10-16% 損耗會變得越來越可以忽略。這個技術參數的變化，會讓「架構上的取捨」重新被評估。

**第三，「Skills」模式的興起代表另一條路的可能。** Skills 的核心特性是「只在需要時才載入」，這直接解決了 MCP 靜態消耗 context 的問題。如果 Skills 生態持續成熟，它可能會成為另一個有力的標準——尤其是在 Anthropic 主导的生态中。

MCP 的故事，其實是 AI 產業標準形成過程的一個縮影：不是最好的技術贏，而是「最多人用 + 最多模型支援」的組合贏。OpenAI 團隊說「實際上幾乎每家公司都在建立 MCP 伺服器」——這句話的重量，超過了任何技術規格的爭論。

**MCP 作為一個術語是否存活，可能不重要。作為讓 AI 代理能夠發現和呼叫工具的類別，它只會越來越重要。**
