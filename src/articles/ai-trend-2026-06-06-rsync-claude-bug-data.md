六月第一週的 Hacker News 首頁，一篇文章連續三天佔據高位、累積 287 點與 284 則留言，標題是：

> **Did Claude increase bugs in rsync?**

這篇文章不是抱怨、不是爆料、不是又一輪的「AI 寫爛程式碼」論戰——它是一份完整的統計分析。從 v2.4.6 到 v3.4.3 共 36 個 rsync 版本、用 Qwen3 35B 為每個 bug 打 0–100 的 severity 分數、跑了精確排列檢定（exact permutation test）與費雪精確檢定（Fisher's exact test）——結論是：Claude 寫的 commit 沒有讓 rsync 變差。**嚴格來說，Claude 版本的 bug 率還略低於歷史平均。**

而這場爭論的故事，必須從十天前那則「無證據的 Mastodon 貼文」說起。

## 一、暴動的起點：一則「有相關性就夠了」的貼文

五月底，獨立開發者 Jeremiah Fieldhaven 在 Mastodon 上發了一則貼文——標題翻成中文大約是「rsync 3.4.3 有數百個 Claude commit」。

貼文本身沒有附 bug 報告、沒有重現步驟、沒有技術分析。它只是把兩件事並列：使用者升級到 v3.4.3 後遇到的某個 regression，以及該版本的 commit log 裡有大量「Co-authored-by: Claude」字樣。從這個巧合，貼文推導出了一個結論：**rsync 變爛了，是 Claude 的錯。**

這則貼文在 Mastodon 上獲得數千個 like 與 boost，58 則回覆來自 32 個獨立用戶。有人主張 rsync 應該被加入「open-slopware 黑名單」、有人說這證明 LLM 就是「認知投降」（cognitive surrender）、有人開始肉搜 rsync 維護者 Andrew Tridgell。

接著，Hacker News 上出現了「Rsync 3.4.3 has hundreds of Claude commits」的討論串（123 點、81 則留言）。輿論從邊緣社群擴散到主流開發者圈。

**五天後**，爭論在 GitHub 上凝結成一個具體的爆點——有人開了一個 issue，標題是：

> **"Please Do Not Vibe Fuck Up This Software"**

issue 裡只附了 Mastodon 貼文的截圖。沒有 bug 重現、沒有技術分析、沒有提問。只有立場。

這個 issue 的留言區從「這軟體是免費的，不喜歡就 fork 或滾」一路升溫到「威脅 fork 到 openrsync」「威脅對維護者進行人身攻擊」——根據後續報導，留言區裡最激烈的內容後來被刪除，但部分已被存檔。

Andrew Tridgell 是 rsync 這個三十年老牌的維護者、Samba 共同作者、也是開源社群備受敬重的工程師。他面對的不是一個技術問題，而是一場社群公審。

## 二、官方回應：Tridge 的 Medium 長文

Tridge 在六月初發了一篇 Medium 長文回應整起事件。核心訊息是：

> 「過去幾個月我被一連串資安報告淹沒——其中很多是 AI 生成的。當這波攻擊越來越密集，我意識到我必須大幅強化 rsync 的防禦：我們需要更完整的測試套件、覆蓋率分析、在更多平台上的 CI 測試、主動的漏洞掃描（這樣我才能在別人發現之前先找到一些）、以及大量的 defense-in-depth 強化。這些都是巨大的工作量。」

換句話說，**v3.4.3 的 commit 數量暴增不是因為 Tridge 在 vibe coding，而是因為 AI 生成的漏洞報告逼得他必須大規模重構資安相關程式碼。** Tridge 自己並沒有否認使用 Claude——他用 Claude 來協助處理這波 AI 生成的漏洞洪流。

他在文中一段話被廣為引用：

> 「對於那些說『我是某大學博士，我告訴你 LLM 只是隨機工具，會編造一切，使用它世界就會完蛋』的人，我在這裡告訴你們：你們已經過時了。軟體工程的世界在過去幾個月已經戲劇性地改變。在 AI 報告的洪流中維持軟體安全的世界，在過去幾週內已經徹底改變。你去年學的這些東西，可能跟另一個星球有關……」

但這篇回應，沒有平息爭論。

## 三、那份改變討論走向的統計分析

爭論的轉折點，是 Alexis Purslane 這個週末發布的網頁：**alexispurslane.github.io/rsync-analysis/**

這篇文章不是情緒宣洩、不是立場表述——它是一份完整的、可重現的（reproducible）統計分析。

### 核心指標

Purslane 沒有只數「bug 數量」（因為一個 typo 和一個 CVE 不應該等量齊觀），而是定義了一個更細緻的指標：

> **sev/10c = (Σ severity/100 ÷ total_commits) × 10**

每個 bug 由 Qwen3 35B（一個小型開源模型）以「資深可靠性工程師」的身份評分，0–100 分（90–100 是資料損毀、70–89 是 crash 或備份中斷、50–69 是 feature regression、30 以下是 UI/文件小問題、0 是 feature request 或垃圾留言）。評分時使用 temperature=0 + JSON schema 結構化輸出，確保可重現。

### 三層統計檢定

**1. Exact Permutation Test（精確排列檢定）**

把所有 36 個版本排成序列，問：如果隨機抽 2 個版本（模擬成「Claude 版本」），抽到的那 2 個版本的平均 sev/10c，比真實 Claude 版本（v3.4.2 + v3.4.3）還要糟的機率是多少？

> **答案：46%**

也就是說，**「Claude 版本比較糟」這個假設的可信度，大約跟丟銅板一樣。** 272/595 的隨機組合都比 Claude 版本差。

**2. Fisher's Exact Test（費雪精確檢定）**

把所有版本用歷史中位數（0.74 sev/10c）切成兩半，問：Claude 版本是否比非 Claude 版本更容易落在「比較糟」的那一半？

> **答案：74%**

勝算比（odds ratio）1.06，幾乎是 1:1。**Claude 版本落進上半部的機率，與隨便挑一個版本完全一樣。**

**3. Wald–Wolfowitz Runs Test（連檢定）**

有人反駁說：早期 rsync 比較穩定、近期 v3.x 比較活躍，會不會是 regime 差異把 Claude 推成了假異常？這個檢定問：歷史上 v3.x 各版本之間有沒有明顯的叢集（clustering）現象？

> **答案：p=0.060**

邊際顯著，但不到 0.05 的傳統閾值——而且 Purslane 直接補做了 v3.x 內部的對比，**結果 Claude 版本在 v3.x 內部依然不顯眼。**

### 額外發現：commit 量與 bug 量的解耦

分析裡有兩個容易被忽略但極具說服力的數字：

| 指標 | Claude 版本 | 非 Claude 版本 | p-value |
|------|------------|---------------|---------|
| 平均 commit 數 | 42 | 185 | 88% |
| 平均行數變更 | 3,756 | 696 | **5%** |
| 平均 severity 加權 bug | 5.6 | 14.9 | 77% |

**Claude 版本改動的行數是歷史平均的 5 倍，但 severity 加權的 bug 數反而略低。** 換句話說，Claude 寫了更多程式碼、改了更多地方，但 bug 密度並沒有上升——這正是「認知投降派」最不樂見的數據。

### 那個最打臉的對照組

Purslane 在文章最後放了一個他自稱「最喜歡的部分」：

> **v3.4.1（沒有 Claude）有史上最高的 bug 率：39.39 sev/10c。** 9 個 commit 帶 59 個 bug，是個 release 隔天的 hotfix。

但當時「沒有人注意到」——沒有 AI 可以怪罪、沒有 GitHub issue 累積 300 則留言、沒有死亡威脅、沒有人威脅要 fork。一個維護者發了個壞掉的版本、修了，就這樣過去了。

> 「唯一讓 v3.4.3 變得特別的，是有個每個人都已經決定要討厭的敵人。」——Alexis Purslane

## 四、社群反應：284 則留言裡的五種聲音

Hacker News 的留言區呈現了這場爭論的所有典型樣態：

**第一種：技術驗屍派**。GodelNumbering 直接翻 git log，找到一個具體的 Claude 寫的 commit：把 `malloc` 改寫為 `calloc`，理由是「strict upgrade」——但對於大塊或遞迴配置，calloc 會造成顯著的額外成本，後來被 revert。連 revert 的 commit 訊息看起來也像 LLM 寫的。

**第二種：數量懷疑派**。wolletd 指出：rsync 過去兩個月的 commit 數，已經逼近過去兩年的總和。「當有人對 AI 用得興奮、變得……草率時，就會出現這種東西。」

**第三種：維護者同情派**。fooker 引用 Tridge 的 Medium 文章，認為這是必然趨勢——「你的工具被 AI agent 24/7 找漏洞，你就需要等量級的防禦。這是新的軟體工程科學的一部分，雖然還在摸索中。」

**第四種：方法論質疑派**。少數人質疑 Purslane 的分析——例如批評「Claude vs Non-Claude」這種二元切分太粗暴，或批評 severity 評分用 LLM 來打分本身就有 circularity 風險。

**第五種：陰謀論派**。少數留言則已經進入「Tridge 是不是被 Anthropic 收買了」的方向——這種言論沒有具體證據，但顯示了當公共討論失去數據錨點時，會自然往陰謀方向滑落。

值得注意的是，**前四種聲音大致構成了健康的工程辯論；第五種則是敘事失控的訊號。** 一份嚴謹的統計分析能壓制前幾種聲音的極端版本，但很難改變已經進入陰謀論框架的人。

## 五、為什麼這場爭論值得開發者社群認真看待

把這件事當成「又一場 AI 口水戰」會錯過它真正重要的部分。

### 1. 敘事的成本：免費的憤怒，昂貴的證據

Mastodon 貼文只需要幾分鐘就能寫出來，它引發的漣漪——58 則回覆、跨平台擴散、GitHub issue、媒體報導、維護者需要寫數千字回應——卻是真實的成本。**Purslane 的分析則需要蒐集 36 個版本的 bug 資料、設計評分標準、跑三層統計檢定、寫出可重現的方法論。** 社群中，產出懷疑的成本遠高於產出恐慌。

### 2. 「相關性誤判」是 AI 時代的職業病

這次爭論的起點是一個看起來很合理的觀察：「我升級到 v3.4.3 後出問題了，這個版本有 Claude commit。」對個別使用者來說，這是一個完全可理解的歸因。**但當我們把視角拉高到 36 個版本，Claude 版本和「比較糟」之間就完全沒有統計關聯。** AI 工具讓我們可以快速生成看似有理的敘事，但這同時也讓我們更容易在沒有證據的情況下做因果歸因。

### 3. AI 寫程式 ≠ AI 寫爛程式：行數暴增才是真訊號

Purslane 的分析裡被低估的一個數據是：Claude 版本改動的行數是歷史平均的 5 倍，但 bug 密度沒有上升。**這告訴我們的不是「AI 寫程式碼不可靠」，而是「AI 讓人敢動更多東西」。** 真正的風險不是 bug 率變高，而是 commit 規模放大後，code review 的負擔變重。當一個 PR 從 200 行變成 3,700 行時，任何審查流程都會被嚴重挑戰。

### 4. 維護者變成公審對象的常態化

rsync、Samba、curl、OpenSSL——這些開源基礎設施的維護者，過去一年承受了「AI 工具生成的資安報告」與「AI 工具是否可靠」的雙重壓力。Tridge 不是個案，他是這個時代的樣板：**當 AI 同時改變了攻擊面與防禦工具，基礎設施的維護者必須在聚光燈下做判斷，而社群用敘事而非數據來評判他們。**

## 六、給開發者的具體建議

**當你看到「X 工具因為 AI 變差」的爆料時，先問三個問題：**
- 有沒有給出 baseline？（對照組是什麼？bug 率相對歷史是上升、持平、還是下降？）
- 有沒有給出檢定方法？（p-value、effect size、信心區間）
- 有沒有混淆「commit 數量變多」與「bug 率上升」？

**當你用 AI 寫程式碼時，記得 Purslane 數據的真正訊號：**
- 真正會爆炸的不是 bug 密度，而是 commit 規模。準備好相應的 review 流程。
- 統計檢定需要時間與數據；不要在樣本數 < 30 的情況下下結論——v3.4.3 的樣本數只有 1。

**當你在社群裡批評一個開源專案時，記得 v3.4.1 的教訓：**
- 沒有 AI 可以怪罪的版本，bug 率 39.39，沒有人在乎。
- 有了 AI 可以怪罪，bug 率 3.29，社群暴動。
- 你的批評強度應該對齊證據強度，而不是對齊情緒強度。

## 七、結論：證據的成本是文明的成本

rsync 暴動是 AI 時代一個微型的、但完整的縮影：社群如何用一個無證據的相關性觀察發起一場敘事戰爭，維護者如何在被圍攻的同時繼續修 bug，最終如何被一份紮實的、可重現的數據分析拉回理性的對話。

這場爭論的結果不是「AI 寫程式碼沒問題」或「Claude 是好工具」——這些都不是 Purslane 試圖證明的。**他想證明的是一個更簡單的命題：「針對 Claude 的指控，在現有資料下，找不到任何統計上的支持。」**

在 AI 敘事充斥開發者社群的 2026 年，**這份「找不到支持」本身，就是最重要的發現。** 因為它意味著：那些看起來很有說服力的 AI 恐慌言論，背後往往只有一兩個逸事、一則 Mastodon 貼文、一個截圖。當我們願意投入時間去檢驗，它們常常就會像 v3.4.3 的 bug rate 一樣，落到歷史分佈的中段。

開源基礎設施的維護者正在一個極度艱難的環境中工作——AI 攻擊、AI 防禦、AI 工具、AI 恐慌四股力量同時拉扯。**他們最不需要的，是社群在沒有數據的情況下，把「相關性誤判」升級為「人格謀殺」。**

下次你想寫一篇「X 工具因為 AI 變爛了」的文章時，先花 30 分鐘做 Purslane 那樣的事——把所有版本、所有 commit、所有 bug 都攤開來看。**你會發現，「敘事」和「證據」之間的距離，往往大得驚人。**
