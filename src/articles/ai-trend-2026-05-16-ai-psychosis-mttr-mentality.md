## 導言

五月中旬，Mitchell Hashimoto 在 X 上寫了一篇不到三百字的貼文。

他說他「強烈相信」：此刻有「整間公司」正在承受嚴重的 AI 精神病，而這個判斷建立在一個具體的觀察上——這些公司和從業者，正在用一種他稱之為「幾乎絕對的 MTTR 至上主義」的方式運作。

MTTR，Mean Time To Recovery，平均修復時間。這個指標在雲端基礎設施時代被用來證明：「沒關係，系統會快速修復，所以可以更快部署。」這個邏輯在雲端時代已經引發過一次清算——我們稱之為「雲端自動化帶來的可用性崩潰」——但現在，它帶著 AI 的包裝，正在軟體開發的全領域重新上演。

這篇貼文獲得了 738 個 HN 星星，以及 321 條留言。這個數字本身不是重點。重點是：為什麼一個基礎設施領域的老問題，換了一個術語包裝，就足以讓整個軟體產業重新陷入同一個陷阱？

---

## MTTR 至上主義的邏輯

讓我們先把 Mitchell 描述的這個「AI 精神病」拆解清楚。

它的核心邏輯是這樣的：當 AI agent 能夠以人類無法企及的速度和規模修復問題時，「事後修復」的成本就會大幅下降。這意味著，「事前預防」的重要性相對降低——你可以更快速地部署、更寬鬆地接受缺陷、更少地依賴冗餘設計。

這個邏輯在某些維度上是成立的。如果 AI agent 真的能夠：
- 及時發現缺陷
- 準確理解問題根因
- 快速產出正確的修復
- 並且確保修復不會引入新的問題

那麼 MTTR 的降低，確實可以在一定程度上抵銷 MTBF（平均故障間隔時間）下降的風險。

問題在於：這四個「如果」的假設，每一個都比看起來更脆弱。

---

## 第一層脆弱：AI 的修復不是真正的修復

當人類工程師修復一個 bug，過程通常包含：理解系統的全域狀態、評估修復的 side effect、在上下文脈絡中權衡各種取捨、然後產出一個符合系統整體架構方向的變更。

當 AI agent 修復一個 bug，過程是：根據 prompt 和 context 生成一段看起來合理的程式碼，並且在有限的測試範圍內驗證。

這兩種修復的本質是不同的。一個是系統性的理解，一個是模式匹配與生成。這並不是說 AI agent 的修復沒有價值——而是說，把 AI 的 MTTR 等同於人類工程師的 MTTR，是一個類比謬誤。

HN 留言中有一位開發者的描述特別精準：他觀察到，當一個 AI coding agent 快速「修復」了十個 bug，系統的整體複雜度通常會增加，而不是減少。因為這些修復是局部最優的，它們沒有被放在系統全域的脈絡下去評估——而這，正是所謂「技術債務」最常見的生成方式。

這裡有一個具體的場景：假設一個電商系統的庫存模組原本的設計邏輯是「同步庫存、直接扣減」。人類工程師在重構時，會理解這個設計背後的業務假設（低併發、庫存充足），並在重構時保留這些語義約束。但當 AI agent 快速修復多個相關 bug 時，它可能會在不理解原始設計意圖的情況下，引入「看起來修復了這個 bug、但實際上破壞了那個假設」的變更。這種破壞在當下的測試環境中不會被發現，因為測試案例通常只驗證「這個 bug 是否被修復」，而不驗證「這個修復是否引入其他假設的不一致」。

---

## 第二層脆弱：錯誤率的不對稱性

AI agent 的錯誤有兩種：可以被快速發現的，和不能的。

可以被快速發現的錯誤，通常是語法層級的、型別錯誤、或者是明顯的邏輯問題。這些錯誤的 MTTR 確實很短。

不能被快速發現的錯誤，通常存在於更深的層次：架構設計的不一致、邊界條件的處理邏輯錯誤、或者是不同模組之間因為修復而引入的整合問題。這些錯誤的 MTTR 不僅不短，而且往往需要等到系統在 production 環境中運行一段時間、累積了足夠的使用者行為資料之後，才會浮現。

問題在於：當一個組織採用「MTTR 至上」的邏輯時，它傾向於用「本地指標」來評估健康狀態——修復速度、錯誤數量、測試覆蓋率——而忽略了「全域指標」：系統架構的完整性、風險暴露面積、錯誤之間的關聯性。

這正是 Mitchell 所說的：「系統可以看起來健康，同時全域正在變得不可理解。」

舉一個具體的例子：一家金融科技公司在引入 AI coding agent 後，bug 回報率在六個月內下降了 40%。看起來是巨大的進步。但進一步分析會發現：bug 回報率下降的同時，系統的平均事故間隔時間（MTBF）實際上也在縮短——只是這個指標從來沒有人追蹤。那些「被修復」的 bug，其實只是被局部地修復了，但系統的整體脆弱性在上升。這個脆弱性在平時不會顯現，但在峰值流量或異常事件的觸發下，會以一種連鎖反應的方式爆發出來。

---

## 第三層脆弱：回顧機制的消失

傳統軟體工程中，「事後檢討」是一個被高度重視的環節。當一個重大事故發生，整個團隊會進行 RCA（Root Cause Analysis，根因分析），然後從中提取系統性的改進方向。

但當 bug 被 AI agent 快速修復之後，這個回顧的動機就會減弱。「反正修好了」的思維，會抑制「為什麼會壞」的探索。而這種探索，恰恰是系統性改進的起點。

留言中有一位基礎設施工程師提到，他在雲端時代親身經歷過同樣的循環：一開始，雲端提供的高可用性和快速擴展能力，讓團隊可以更快速地部署、更快速地修復。「反正雲端會處理」的假設，讓許多原本需要在部署前解決的問題，被「修復就好」取代了。但這個循環最終會到達一個臨界點：當系統的複雜度超越了任何人能夠完整理解的邊界，無論雲端多麼強大，修復的成本都會開始指數成長。

現在，AI agent 的出現，正在把同樣的循環加速重演。

一個具體的差異在於：雲端時代的自動化，主要是「基礎設施」的自動化——伺服器、網路、儲存。這些元件的行為雖然複雜，但至少是確定性的。而 AI coding agent 的自動化，是「邏輯」的自動化——商業規則、資料處理流程、使用者介面互動。這些邏輯的不確定性更高，而且往往存在於文件沒有記載、只有程式碼本身才知道的微妙假設中。

## 第四層脆弱：測試覆蓋率的幻覺

Mitchell 在貼文中特別點出了一個現象：「測試覆蓋率在上升，同時語義理解在下降。」

這句話的精準程度，值得反覆咀嚼。

測試覆蓋率是一個量化指標，它測量的是「有多少行程式碼被測試執行過」。但它不能測量的是「這些測試是否真正捕捉了有意義的系統行為」。

當 AI agent 快速生成大量的單元測試時，覆蓋率數字會很好看。但這些測試的設計品質，取決於 AI 對系統預期行為的理解——而 AI 的理解，是從 prompt 和現有程式碼中推斷出來的，不是從「系統應該做什麼」的語義層面建立的。

這意味著：覆蓋率可以很高，但測試的內在品質可以很低。一個被高覆蓋率掩蓋的低品質測試套件，比沒有測試更危險——因為它給予了一種虛假的安全感，讓團隊在「測試通過」的信心下，快速部署，然後在 production 環境中遇到那些「覆蓋率數字從來沒有告訴過你」的錯誤。

舉例來說：一個支付系統可能有一千個測試案例，覆蓋率達到 95%。但這 95% 的覆蓋率，可能從來沒有測試過「當支付失敗時，庫存鎖定是否正確釋放」這個跨模組的語義約束。AI agent 生成測試時，會根據現有的程式碼結構生成覆蓋那些程式碼路徑的測試，但不會根據「系統應該遵守的業務規則」生成測試——因為那些業務規則，往往存在於人類工程師的經驗中，而不是存在於任何具體的程式碼或文件裡。

---

第五層脆弱：專業判斷能力的鈍化

這也許是最少被討論，但影響最深遠的一層脆弱。

當工程師不再需要親手修復每一個 bug，他們面對系統問題的頻率會下降。而這種頻率的下降，會逐漸鈍化他們對系統行為的直覺——那種在長期浸泡於系統中才能建立的、難以言傳的「第六感」。

一個有二十年經驗的資深工程師，他對系統的掌握，很大程度上來自於他親身經歷過的那些「差一點就出事」和「這次終於出事」的時刻。這些時刻的積累，給予他一種在問題爆發之前就能嗅到風險的能力。

但當 AI agent 在第一時間就修復了那些「差一點就出事」的情況，這些學習機會就被繞過了。工程師看到的系統，永遠是「運作正常」的，而那些本應提醒他們「這裡有潛在問題」的信號，都被靜音了。

這不是一個可以立即量化的問題，但它是一個會隨著時間累積的問題。當組織的工程團隊在三年後失去了這種直覺，而 AI agent 的限制變得更加明顯時，這個代價才會被感受到——只是那個時候，可能已經太晚了。

---

## 為什麼這個問題在 AI 時代更嚴重

如果這個問題在雲端時代就存在，為什麼現在更值得擔心？

因為 AI agent 的介入，改變了幾個關鍵參數。

**第一，速度。** 人類工程師修復一個 bug 需要專注的時間、上下文切換的心理成本、以及對系統狀態的深入理解。AI agent 的修復速度，壓縮了這個過程到秒級。當修復變得如此快速，MTTR 的降低是真實的——但這也意味著，「快速修復」的循環可以以更高的頻率運轉，系統複雜度可以在更短的時間內累積到臨界點。

**第二，顆粒度。** 過去，雲端基礎設施時代的自動化，主要是「基礎設施」的自動化。但 AI coding agent 的自動化，是「邏輯」的自動化。當邏輯層面的變更可以如此快速地產生和部署，系統的不可理解性，會比基礎設施時代以更快的速度成長。

**第三，責任歸屬的模糊化。** 當一個 bug 被 AI agent 修復，誰應該負責？這個問題目前沒有好的答案。而責任歸屬的模糊，會進一步抑制那些本應在事後發生的學習和改進。

**第四，規模。** AI agent 可以同時處理多個任務，而人類工程師不行。這意味著，在同樣的時間範圍內，AI 可以累積更多的系統變更、更多的技術債務、以及更多的「看起來沒問題但實際上有問題」的角落。

## 歷史不會重演，但會押韻

Mitchell 在貼文中提到了他親身經歷過的「雲端自動化時代的基礎設施清算」。那個時代的教訓是：當 MTTR 成了唯一的指標，MTBF 就會被忽視；而當兩者都被忽視時，最終得到的，是一個「自我強化的災難機器」——看起來運作良好，實際上正在累積不可見的風險，直到有一天，這些風險以一種連鎖反應的方式集體爆發。

現在，這個歷史正在 AI coding 的領域，以不同的術語，重新上演。

但這一次，有一個關鍵的不同：AI 時代的軟體系統，正在比雲端時代更加深度地嵌入業務邏輯。當系統的不可理解性達到臨界點時，它的後果，不會只是「服務中斷」，而可能是「商業邏輯的不可預測行為」，以及隨之而來的客戶信任、法律責任、和品牌損害。

更具體地說：在雲端時代，一個被自動化掩蓋的基礎設施問題，可能只影響系統的可用性。但在 AI coding 時代，一個被快速修復掩蓋的邏輯問題，可能會影響系統的正確性——而正確性的問題，往往比可用性的問題更難被事後發現，也更難被事後修復。

## 對開發者的具體建議

這篇文章不是要主張「AI coding 是壞的」，也不是要製造那種「AI 會摧毀一切」的恐慌論述。恰恰相反——AI coding 的能力是真實的，它的價值也是真實的。問題不是 AI 本身，而是應用 AI 的方式。

這裡有幾個具體的觀察點：

**理解 AI 修復的本質限制。** AI agent 的修復是局部最優的，不是全域最優的。這個限制不會因為模型的能力提升而消失——它是由 AI 的工作方式決定的。因此，在使用 AI 修復的同時，仍然需要人類工程師從系統全域的視角，進行定期的架構檢視和債務盤點。具體的頻率，建議每兩週进行一次系統性的架構回顧，而不是只在事故發生後才介入。

**建立不依賴 MTTR 的品質文化。** MTTR 是補救措施，不是品質策略。當一個組織開始用 MTTR 的降低，來合理化測試品質的下降或架構投資的減少時，這是一個危險的信號。真正的品質，來自於對系統行為的深入理解，來自於冗餘設計和失敗模式的預先分析，來自於那些看起來「沒有直接價值」的前置作業。一個實用的做法是：在團隊的 KPI 中，明確地加入「系統可理解性」或「技術債務清理進度」這樣的維度，而不是只看「MTTR」和「bug 數量」。

**保留回顧的機制。** 當 AI 快速修復了問題，團隊仍然需要問「為什麼會發生這個問題」，而不是只滿足於「問題已經修復」。這個問「為什麼」的過程，是系統性學習的唯一路徑。建議建立一個「每季技術回顧」的機制，专门审视那些「被快速修復」的 bug，看看它们之间是否有共同的根因，以及这些根因是否指向系统性的设计问题。

**不要放棄親手處理的機會。** 對於關鍵系統模組，仍然應該保留人類工程師親手處理的時間和空間。這不是效率最優的選擇，但它是保留「系統直覺」的唯一方式。具體的做法，可以是每週安排一定比例的「人類優先」任務——在這些任務上，明確要求人類工程師親自分析和修復，而不是交給 AI agent。

## 結語

Mitchell Hashimoto 的貼文引發的討論，觸及了一個古老的工程問題：我們如何在一個複雜系統中，保持對系統行為的真正理解，而不只是在局部指標上看起來一切正常？

這個問題在雲端時代被提出過，在微服務時代被討論過，在 DevOps 時代被重新定義過。現在，它帶著 AI coding 的包裝，再次成為焦點。

歷史不會完全重演，但會押韻。上一次我們學到的教訓是：「自動化可以創造一個非常有彈性的災難機器。」這一次，我們也許會學到同樣的 lesson，只是代價可能更大。

未來三個月，值得持續觀察的是：這些「AI 精神病」的公司，有多少會在系統複雜度到達臨界點時，開始出現集體性的事故？以及，當這個時刻到來時，整個產業會不會真正開始認真面對「MTTR 不是一切」這個命題？

還是說，我們會繼續用更快的 AI agent，來解決由 AI agent 快速修復所累積的技術債務，直到這個債務的規模，超越任何工具的能力邊界？

這個問題現在沒有答案。但它的答案，會決定 AI coding 這場革命的最終代價。
