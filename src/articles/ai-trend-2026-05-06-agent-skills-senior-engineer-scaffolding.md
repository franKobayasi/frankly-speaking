## 導言

上週 Hacker News 出現了一個獲得 367 顆星的連結，指向 Addy Osmani 的新專案：Agent Skills。短短几天，GitHub 上的星星數從零爬升到兩萬七千多顆。

這個數字本身不說明什麼。HN 的流量可以讓任何一個時機湊巧的專案爆紅。但當你實際去讀這個專案的設計文件，會發現它試圖回答的問題，比表面看起來要深刻得多：

**為什麼 AI  coding agent 那麼擅長寫 code，卻那麼不擅長「工程」？**

---

## Agent Skills 在做什麼

簡單來說，Agent Skills 是一套給 AI coding agent 用的「資深工程師 SOP」。

它以 markdown 檔案的形式存在，每個 skill 包含一個工作流程：步驟、檢查點、具體的退出標準。這不是「關於測試你應該知道的一切」那種兩千字的論文，而是一個 agent 可以實際執行的序列。

Addy Osmani 在他的部落格文章中用一個對比說明了這個差異：

> 把測試最佳實踐的長文塞进 agent 的 context，它會讀完、產出看起來像樣的文字，然後跳過真正的測試。但如果放進去的內容是「先寫一個會失敗的測試、執行它、確認它失敗、寫最小代碼讓它通過、再確認它通過」——那 agent 就有事可做，你也有東西可以驗證。

這個 distinction 就是整個遊戲的關鍵。Agent Skills 裡的二十個 skills 圍繞六個生命週期階段組織：Define（定義）、Plan（規劃）、Build（實作）、Verify（驗證）、Review（審查）、Ship（發布）。每個階段都有對應的 slash command 和 checkpoint 機制。

背後的邏輯並不新穎——就是任何健康工程組織都在執行的 SDLC，只是用不同的詞彙包裝。Agent Skills 的價值在於，把這套流程「翻譯」成 AI agent 可以理解、可以執行的形式。

---

## 為什麼這件事現在變得重要

### 短視是預設行為

AI coding agent 的預設行為，是走最短路徑抵達「完成」。你要一個功能，它就寫功能。它不會問你有沒有 spec，不會先寫測試，不會考慮這個變更是否跨越了信任邊界，也不會檢查 PR 拿出來長什麼樣。

這不是 agent 的缺陷。這是訓練的信號決定的：「任務完成」就是回饋，「任務完成且 design doc 存在」不是。沒有信號激勵的行為，就是不會發生的行為。

大多數资深工程師花在「不在 diff 裡的工作」的時間—— surfacing assumptions、寫 spec、把工作切成可審查的區塊、選擇無聊但正確的設計、留下證據證明結果是對的——在 AI agent 的世界裡，這些全部消失了。

### 分類戰爭的下一階段

過去一年半，AI  coding tools 的主流戰場是「誰的模型更強」。GPT-4、Claude Opus 3.5、DeepSeek V4 Pro——每次頂級模型的更新都會引發一波討論，各家工具也以「支援最新最強模型」作為主要賣點。

但 Agent Skills 的出現，暗示戰場正在轉移。當底層模型的差異化越來越小（DeepClaude 已經展示了用 DeepSeek 置換 Claude 的可行性，成本只有 1/17），框架層的價值開始被重新看見。

Agent Skills 試圖回答的問題是：**框架應該內建什麼樣的工作流程？** 這個問題，會是下一階段競賽的核心。

## 看不見的工作為什麼看不見

Addy Osmani 在文章中提出一個有趣的觀察：他稱之為「反理性化表格」（anti-rationalization tables）。

每個 skill 都包含一個常見藉口的表格——agent（或者一個疲憊的工程師）可能用來跳過這個流程的藉口，配上書面的反駁。幾個接近原文的例子：

- 「這個變更很小，不需要寫 spec」→ 反駁：spec 的目的不是記錄大變更，而是捕捉假設
- 「反正會有 code review，現在先 merge」→ 反駁：review 的輸入品質決定了 review 的品質

這些藉口聽起來合理嗎？對。因為它們就是合理。它們是每次工程衰敗時真實發生的對話。

LLM 擅長理性化。它們會產生聽起來可信的段落，解釋為什麼 spec 不是必要的，或者為什麼某個變更可以在沒有 review 的情況下合併。但如果把 anti-rationalization 表格放進 context，agent 就有了一個可以直接引用的反駁，而不是自己發明一個。

這個設計模式，不只對 AI agent 有用。對人類團隊也一樣。

---

## 驗證是不可協商的

Agent Skills 的另一個核心原則：每個 skill 都終止於具體的證據。測試通過，建構輸出乾淨，runtime trace 顯示預期行為，reviewer 簽核。

Addy Osmani 的原文說：「從 hearsay 說『看起來沒問題』是不夠的。」

這呼應了最近 Reflex 的 benchmark 發現：同樣的任務，vision agent（電腦使用）需要 53 步、551k tokens，而直接呼叫 API endpoint 只要 8 次 call、12k tokens。兩者都在「完成任務」，但只有後者有明確的驗證機制。

當 vision agent 無法完成任務時——比如說，找到某個名字的客戶、接受他們的所有評論、標記訂單為已交付——它的失敗原因是「頁面沒有顯示全部結果，agent 不知道要去下一頁」。這不是模型的問題，這是框架沒有提供驗證信號的問題。

驗證框架與生成器分離的地方，就是錯誤累積的地方。

---

## 對開發者的意義

Agent Skills 不是銀彈。27K stars 是一個社群信號，不是一個經過大規模生產驗證的解決方案。它目前的二十個 skills 是 Addy Osmani 對「資深工程師 scaffold 應該長什麼樣」的最佳猜測，映射了 Google 內部的工程實踐。

但它觸及的問題是真實的。開發者社群正在用星星投票，告訴工具製造商：我們不只是要「AI 寫 code」，我們要的是「AI 做工程」。

這個需求的具體含義是：

**第一，工作流程會變成一等公民。** 買的不再只是「模型能力」，而是「模型加上它內建的工作流程」。一個每次都走最短路徑的 agent，和一個會先問「有 spec 嗎」的 agent，長期價值差異巨大。

**第二，anti-rationalization 文件會成為團隊資產。** 將團隊的「明知故犯」模式寫下來，是對抗集體慣性下滑的有效機制。不只是給 AI 看的，更是給團隊自己看的。

**第三，驗證成本會被認真計算。** Reflex 的 benchmark 已經把成本數字擺在桌上：45 倍的差異。當企業開始認真比較「看起來完成了」和「有證據證明完成了」之間的代價，工具設計的優先順序會跟著調整。

---

## 結語

Agent Skills 的星星數會繼續上升或穩定，取決於這個專案能不能在大規模使用中站穩。但它揭示的趨勢不會退潮：AI coding agent 正在從「能跑」走向「好用」，而這段距離的中間，是資深工程師數十年累積的沉默紀律。

未來幾個月，值得觀察的是：主流 Agent 框架（Claude Code、Cursor、Copilot）會不會將類似的 workflow 機制內建進去？當框架開始內建「資深工程師 scaffold」，而不是把這些視為可選的加成，AI 開發工具的定義會重新改寫。

這個問題的答案，會比 Agent Skills 本身更有意思。
